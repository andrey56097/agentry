# Multi-Agent Code Review System с Observability и CI/CD

## Общая идея проекта

Система принимает задачу на естественном языке (например, "напиши функцию валидации email с юнит-тестами") и автоматически прогоняет её через команду AI-агентов (Proposer → Critic → Improver), которые пишут, критикуют и улучшают код в реальном git-репозитории. Весь процесс проходит через настоящий CI/CD-пайплайн (тесты, линтер, мутационное тестирование), а не имитацию оценки качества. Всё видно в реальном времени на дашборде с метриками стоимости (токены/$) и качества (score по раундам).

**Питч для собеседования:**
> "Я построил систему, где вместо одного запроса к LLM задача проходит через конвейер специализированных агентов с итеративной само-критикой на основе реальных CI-метрик (тесты, покрытие, mutation score), а не выдуманных оценок. Плюс — полный observability-слой: стоимость и качество каждого вызова модели."

```
      │         │         ┌─────▼──────┐
      │         │         │ Metrics DB  │
      │         ▼         │ (Postgres)  │

      └─────────┴───────►  Dashboard (React)
                          - Live task pipeline (Kanban)
                          - Cost per task/agent
                          - Round-by-round quality score
                          - Budget burn-down
```

## Поток выполнения задачи (end-to-end)

1. Задача создаётся через дашборд → попадает в очередь
2. Оркестратор создаёт git-ветку `task/{uuid}`
3. **Proposer** пишет первую версию кода → коммит "draft: initial implementation" → push
4. Push триггерит **GitHub Actions**: `mvn test`, линтер, Pitest (mutation testing)
5. Оркестратор получает webhook с результатами CI
6. **Critic** анализирует реальные метрики (не выдуманные): % тестов прошло, покрытие, mutation score, ошибки линтера → выставляет оценку и фидбек
7. Если оценка ниже порога (quality gate) → **Improver** переписывает код с учётом фидбека → коммит → push → CI снова триггерится
8. Цикл повторяется, пока не пройден quality gate или не исчерпан бюджет
9. При успехе → PR готов к авто-мерджу (или ожидает human approve)
10. Все шаги (токены, стоимость, latency, score) пишутся в Postgres и отображаются на дашборде в реальном времени

## Этап 1 — Ядро: оркестратор + агенты + логирование (неделя 1)

**Цель:** рабочий бэкенд без UI, с логами в консоль/БД.

- [ ] Spring Boot проект: сущности Task, AgentRun, CodeVersion
- [ ] Интеграция с Claude API (простые вызовы, без стриминга)
- [ ] Реализовать 3 роли через разные system-промпты:
  - Proposer — пишет минимальный рабочий код
  - Critic — анализирует и оценивает (пока на основе текста, без реального CI)
  - Improver — переписывает с учётом фидбека
- [ ] Простой цикл: `while score < threshold: propose → critique → improve`
- [ ] Postgres: таблица для метрик каждого вызова (task_id, round, agent, tokens, cost, latency, score)
- [ ] Task Budget: лимит токенов на задачу, который **агент сам распределяет** внутри цикла (не просто внешний "стоп-кран")

### Концепция Task Budget — единственный рычаг управления стоимостью

Когда агенты работают автономно (без человека, который останавливает их после каждого шага), главный риск — потратить много токенов на задачу, которая этого не стоит. Простое "остановиться при превышении лимита" — это грубый стоп-кран, а не управление. Правильная модель:

- **Вы задаёте лимит** на задачу (например, 4000 токенов) — это единственный параметр, которым вы управляете стоимостью автономного запуска
- **Агент сам решает, как распределить бюджет** внутри этого лимита:
  - потратить больше на первый черновик и меньше на итерации
  - сделать 1 качественный раунд вместо 3 средних
  - остановиться раньше, если решил, что дальнейшие улучшения не окупятся
- Оркестратор не диктует "сколько раундов делать" — он просто **передаёт агенту остаток бюджета** перед каждым шагом (`remaining_tokens: 1850`), и агент/цикл принимает решение с учётом этого остатка

```java
public class TaskBudget {
    int totalLimit;
    int spent;

    int remaining() { return totalLimit - spent; }

    boolean canAfford(int estimatedCost) {
        return remaining() >= estimatedCost;
    }
}
```

Перед каждым раундом Improver/Critic оркестратор передаёт в промпт что-то вроде:
> "Осталось 1850 токенов бюджета на эту задачу. Учти это при решении, продолжать ли улучшение или зафиксировать текущий результат."

Это даёт модели возможность **самой решать trade-off между качеством и стоимостью**, а не слепо гнать раунды до внешнего обрыва по лимиту. На дашборде (Этап 3) это отображается как burn-down бюджета по задаче — видно, сколько потрачено и сколько агент сознательно оставил про запас.

**Результат этапа:** консольное приложение, которое берёт текстовую задачу и выдаёт финальный код + историю раундов в БД.

## Этап 5 — Независимые агенты и выбор моделей (неделя 5, опционально)

**Цель:** каждый агент (Proposer/Critic/Improver) может работать на своей модели/провайдере, и это можно выбирать через UI, а не хардкодить в коде.

### Идея

Агент описывается декларативно (конфигом), а не жёстко привязан к конкретному API:

```yaml
agents:
  - id: proposer
    role: "Proposer"
    provider: anthropic
    model: claude-sonnet-5
    system_prompt: prompts/proposer.txt
    temperature: 0.7

  - id: critic
    role: "Critic"
    provider: openai
    model: gpt-5
    system_prompt: prompts/critic.txt
    temperature: 0.2

  - id: improver
    role: "Improver"
    provider: anthropic
    model: claude-haiku-4-5
    system_prompt: prompts/improver.txt
```

### Задачи этапа

- [ ] Абстракция провайдера — единый интерфейс поверх разных LLM API:
  ```java
  public interface ModelProvider {
      LLMResponse complete(String systemPrompt, List<Message> history, double temperature);
  }

  public class AnthropicProvider implements ModelProvider { ... }
  public class OpenAIProvider implements ModelProvider { ... }
  public class OllamaProvider implements ModelProvider { ... } // локальные модели
  ```
  Каждый провайдер инкапсулирует свой формат запроса, но возвращает единый `LLMResponse { text, inputTokens, outputTokens, costUsd }`.

- [ ] Реестр агентов вместо хардкода ролей в коде:
  ```java
  public class AgentDefinition {
      String id;
      String role;
      ModelProvider provider;
      String model;
      String systemPromptTemplate;
      double temperature;
  }
  ```
  Оркестратор запрашивает текущую конфигурацию роли ("кто сейчас Critic?"), а не использует зашитую в код привязку.

- [ ] UI на дашборде — выбор модели для каждой роли перед запуском задачи:
  ```
  Proposer:  [Claude Sonnet 5 ▾]
  Critic:    [GPT-5 ▾]
  Improver:  [Claude Haiku 4.5 ▾]
  ```
- [ ] Пресеты конфигураций (например, "дешёвый прогон" — все агенты на Haiku/mini-моделях, "качественный прогон" — Sonnet/Opus/GPT-5)
- [ ] Расширить таблицу метрик:
  ```sql
  ALTER TABLE agent_runs ADD COLUMN provider VARCHAR;
  ALTER TABLE agent_runs ADD COLUMN model VARCHAR;
  ```
- [ ] Дашборд: сравнение стоимость/качество по моделям (A/B-аналитика)

### Зачем это нужно, а не просто "фича ради фичи"

- **A/B сравнение моделей на одной задаче** — готовый инсайт для презентации: "GPT-5 как Critic находит на 20% больше багов, чем Claude Haiku, но стоит в 3 раза дороже"
- **Оптимизация стоимости**: дешёвая модель для черновика (Proposer), дорогая — для финальной критики
- **Отказоустойчивость**: при недоступности одного провайдера — fallback на другой

**Результат этапа:** система превращается из "заточенной под одну модель" в гибкую платформу, где выбор модели под роль — часть эксперимента, а не архитектурное ограничение.

## Этап 6 — Фоновая очистка памяти (Autodream, опционально)

**Цель:** агенты не раздувают контекст до бесконечности — накопленные факты о проекте периодически чистятся, сливаются и сжимаются фоновым процессом между сессиями.

### Проблема, которую решает

Со временем Critic/Improver накапливают контекст о проекте (стиль кода, прошлые договорённости, архитектурные решения). Если просто добавлять всё в system prompt — это:
- раздувает токены (а значит и стоимость каждого вызова)
- создаёт противоречия ("раньше решили делать так, а теперь иначе")
- снижает качество ответов, потому что модель должна выбирать между устаревшим и актуальным

### Идея

Отдельный фоновый процесс (не блокирует основной pipeline задач), который запускается между сессиями или по расписанию и приводит память агента в порядок.

### Схема хранения памяти

```sql
CREATE TABLE agent_memory (
    id UUID PRIMARY KEY,
    fact TEXT,
    confidence INT,        -- 0-100, насколько агент уверен в факте
    tags TEXT[],            -- например: ['релиз', 'план']
    source_task_id UUID,    -- из какой задачи появился факт
    created_at TIMESTAMP,
    superseded_by UUID NULL -- ссылка на факт, который его заменил
);
```

### Агент-куратор (Memory Curator)

Отдельная роль, работающая не в основном pipeline задач, а по расписанию (cron) или после N завершённых задач:

```
1. Достать все активные факты (superseded_by IS NULL)
2. Отдать их LLM с промптом: "найди дубликаты, противоречия, устаревшее"
3. Получить решение по каждой группе:
   - DELETE  — факт больше не актуален
   - MERGE   — несколько фактов слить в один более общий
   - COMPRESS — длинный факт сжать в компактную формулировку
   - KEEP    — оставить как есть
4. Применить изменения, старые записи не удалять физически,
   а помечать superseded_by (для истории/аудита)
```

### Задачи этапа

- [ ] Таблица `agent_memory` в Postgres (схема выше)
- [ ] Запись фактов в память как побочный эффект работы агентов (например, Critic решил "клиент предпочитает короткие письма" → сохраняется как факт с тегами)
- [ ] Curator-агент: отдельный scheduled job (cron), не участвующий в основном Proposer-Critic-Improver цикле
- [ ] Метрика на дашборде: "было N записей → стало M" (как на референсе: "было 8, стало 4"), плюс разбивка "удалено / слито / сжато"
- [ ] Кнопка "Запустить Autodream сейчас" — ручной триггер вне расписания
- [ ] Простая визуализация "до/после" (например, столбчатая диаграмма суммарного объёма памяти в токенах до и после очистки)

### Как это связывается с остальной системой

- **Бюджет**: меньше токенов в system prompt на каждую задачу → прямая экономия, видно на dashboard бюджетов задач
- **Мультиагентность (Этап 5)**: каждый провайдер/модель может иметь свою память — то есть Curator можно тоже гонять на дешёвой модели (Haiku), раз это фоновая некритичная по времени задача
- **Observability**: Curator — такой же агент, как остальные, поэтому его вызовы тоже попадают в общую таблицу метрик (`agent_runs`), просто с `role = 'curator'`

**Результат этапа:** система не деградирует по качеству при длинных сессиях и большом количестве задач — память остаётся компактной и непротиворечивой, а токены не тратятся на устаревший контекст.

## Этап 7 — Портирование на реальный проект (после MVP)

**Цель:** превратить систему из "решателя абстрактных задач в тестовом репо" в **помощника, который вникает в контекст конкретного проекта** и выполняет задачи по заданным в этом проекте шаблонам/конвенциям.

### Фаза 7.1 — Отдельная ветка в существующем репо (первый шаг)

Не отдельный сервис, а расширение уже написанного оркестратора — подключаем его к реальному (не тестовому) репозиторию через отдельную ветку.

- [ ] Создать в целевом репозитории ветку, например `agent-integration` — всё, что делают агенты, живёт в ней, не трогая `main`
- [ ] **Project Context Loader** — компонент, который разово (или инкрементально) читает целевой проект:
  - структура модулей, ключевые файлы (README, pom.xml/package.json)
  - индексация кода в vector DB (RAG-подход — то, что уже в roadmap отдельно)
- [ ] Файл конвенций проекта `PROJECT_CONVENTIONS.md` (аналог `CLAUDE.md`, который ты уже изучаешь) — читается один раз и попадает в system prompt каждого агента:
  ```markdown
  # PROJECT_CONVENTIONS.md

  ## Стиль кода
  - Все DTO — через record, не class
  - Валидация — через Bean Validation, не ручные if

  ## Тестирование
  - JUnit 5 + Mockito
  - Обязательный mutation testing порог: 75%

  ## Архитектура
  - Слоистая: Controller → Service → Repository
  - Никаких бизнес-правил в контроллерах
  ```
- [ ] Задача передаётся в системe в формате, привязанном к этому единственному проекту:
  ```yaml
  task: "Добавить endpoint для отмены заказа"
  constraints:
    - follow: PROJECT_CONVENTIONS.md
    - must_pass: ci/quality-gate
  ```
- [ ] Critic дополнительно проверяет соответствие `PROJECT_CONVENTIONS.md`, а не только тесты/линтер — то есть ругается и на "не баг, но не в стиле проекта"
- [ ] Все PR от агентов идут в `agent-integration`, промежуточный ревью человеком перед мерджем в `main`

**Результат фазы:** система работает над одним реальным проектом (например, твоим текущим pet-проектом), в изолированной ветке, не мешая основной разработке.

### Фаза 7.2 — Выбор проекта (после того, как фаза 7.1 обкатана)

Когда интеграция с одним проектом работает стабильно — обобщаем на множество проектов.

- [ ] Таблица `projects` в БД оркестратора:
  ```sql
  CREATE TABLE projects (
      id UUID PRIMARY KEY,
      name VARCHAR,
      repo_url VARCHAR,
      default_branch VARCHAR,        -- ветка для интеграции агентов
      conventions_path VARCHAR,       -- путь к PROJECT_CONVENTIONS.md
      vector_index_id VARCHAR,        -- ссылка на индекс в vector DB
      created_at TIMESTAMP
  );
  ```
- [ ] UI на дашборде — выбор активного проекта перед созданием задачи:
  ```
  Проект:  [ecommerce-backend ▾]
  Задача:  "Добавить endpoint для отмены заказа"
  ```
- [ ] Память агентов (Autodream, Этап 6) становится **per-project**, а не общей — факты о проекте A не должны просачиваться в контекст проекта B
- [ ] Индексация нового проекта — воркфлоу "Подключить проект": оркестратор клонирует репозиторий, строит vector-индекс, ищет/просит создать `PROJECT_CONVENTIONS.md`
- [ ] Метрики (Этап 3) разбиваются по проектам — можно сравнивать стоимость/качество работы агентов на разных кодовых базах

**Результат фазы:** платформа, а не инструмент под один проект — можно подключить любой репозиторий и получить помощника, работающего по его собственным конвенциям.

### Почему это ценно

Это превращает демо-проект в **реально используемый инструмент** — не просто "я показал, что умею в AI agents", а "я использую это в своей повседневной работе над проектом Y". Это гораздо более сильный аргумент на собеседовании, чем изолированное демо.

## Этап 8 — Авторизация и BYOK (Bring Your Own Key)

**Цель:** система становится многопользовательской — каждый пользователь работает под своим аккаунтом и тратит **свои** API-токены (Claude, OpenAI/Codex и т.д.), а не токены владельца системы.

### Зачем это нужно

Пока система — личный pet-проект, все вызовы идут с одного API-ключа (твоего). Как только систему пробует кто-то ещё (коллеги, потенциальный работодатель на демо, публичный доступ) — нужно разделение: у каждого свой ключ, свой бюджет, своя история задач. Это также убирает риск, что кто-то "сожжёт" твой личный лимит токенов.

### Задачи этапа

**Авторизация**
- [ ] Аутентификация пользователей (простой вариант: email+password или OAuth через GitHub — логично, раз система и так работает с GitHub-репозиториями)
- [ ] Таблица `users`:
  ```sql
  CREATE TABLE users (
      id UUID PRIMARY KEY,
      email VARCHAR UNIQUE,
      github_id VARCHAR,
      created_at TIMESTAMP
  );
  ```
- [ ] Все существующие сущности (`tasks`, `agent_runs`, `projects`) получают `user_id` — данные одного пользователя не видны другому
- [ ] Роли доступа (обычный пользователь / админ) — админ видит агрегированную статистику по всем, обычный — только своё

**BYOK — пользователь платит за свои вызовы**
- [ ] Таблица `api_keys` (ключи хранятся зашифрованными, не в открытом виде):
  ```sql
  CREATE TABLE api_keys (
      id UUID PRIMARY KEY,
      user_id UUID REFERENCES users(id),
      provider VARCHAR,        -- anthropic / openai / etc.
      encrypted_key TEXT,
      created_at TIMESTAMP
  );
  ```
- [ ] Шифрование ключей в БД (например, AES с ключом шифрования из переменной окружения/секрет-хранилища — не хранить в открытом виде никогда)
- [ ] UI: страница настроек, где пользователь вводит свой ключ Claude/Codex — по одному на провайдера
- [ ] `ModelProvider` (из Этапа 5) при вызове LLM берёт ключ конкретного пользователя, а не глобальный ключ системы:
  ```java
  ModelProvider provider = providerFactory.get(agent.getProvider(), user.getApiKey(agent.getProvider()));
  ```
- [ ] Если у пользователя нет ключа для выбранного провайдера — понятная ошибка в UI с просьбой добавить ключ в настройках, а не молчаливый fallback на чужой ключ
- [ ] Бюджет (Task Budget, Этап 1) считается персонально — токены/стоимость трекаются per-user, а не в общий пул

### Важное про безопасность

- Ключи никогда не логировать (даже в debug-логах) и не отправлять на фронтенд после сохранения — только маскированный вид (`sk-ant-...xxxx`)
- Ограничить права хранимого ключа минимально необходимыми (если провайдер это поддерживает — read/write scopes)
- Rate-limiting на уровне пользователя — защита от случайного/злонамеренного исчерпания бюджета через баг в UI

**Результат этапа:** систему можно безопасно показать/дать использовать другим людям — каждый работает под своим аккаунтом, своими ключами и своим бюджетом, без риска для владельца системы.

## Этап 8 — Авторизация и BYOK (Bring Your Own Key)

**Цель:** пользователь подключает свой аккаунт/API-ключ (Claude, OpenAI/Codex и т.д.) и платит за токены сам — система не расходует твой бюджет на чужие задачи.

### Зачем это нужно

Пока система работает только на твоём API-ключе, её нельзя дать попробовать другим людям (друзьям, будущим работодателям, комьюнити) без риска, что кто-то сожжёт весь твой бюджет. BYOK превращает pet-проект в инструмент, которым реально может пользоваться кто угодно.

### Задачи этапа

- [ ] **Аутентификация пользователей** — базовая система логина:
  - email/password или OAuth (GitHub-логин особенно уместен, раз система и так работает с GitHub-репозиториями)
  - JWT/session-based авторизация на уровне Spring Security
- [ ] **Хранилище ключей пользователя** — таблица с зашифрованными API-ключами:
  ```sql
  CREATE TABLE user_credentials (
      id UUID PRIMARY KEY,
      user_id UUID REFERENCES users(id),
      provider VARCHAR,        -- anthropic / openai / ...
      encrypted_api_key TEXT,  -- зашифровано (например, AES, ключ шифрования — отдельно в secret manager)
      created_at TIMESTAMP
  );
  ```
- [ ] Шифрование ключей на уровне приложения (не хранить plaintext даже во внутренней БД) — например, Jasypt или ручное AES-шифрование с ключом из переменной окружения/Vault
- [ ] Каждый вызов `ModelProvider` (из Этапа 5) берёт ключ не из глобального конфига, а из credentials текущего пользователя:
  ```java
  LLMResponse complete(User user, AgentDefinition agent, ...) {
      String apiKey = credentialService.getKey(user.getId(), agent.getProvider());
      return provider.complete(apiKey, ...);
  }
  ```
- [ ] UI: страница "Настройки" → "Подключить свой API-ключ" (Claude / OpenAI / другие) с валидацией ключа (тестовый вызов при сохранении)
- [ ] Метрики (Этап 3) теперь считаются **per-user** — расход бюджета/стоимость видна каждому в своём личном кабинете, а не в общей куче

### Важные нюансы безопасности

- Никогда не логировать ключи в открытом виде (даже в debug-логах)
- Ключ передаётся к провайдеру напрямую с бэкенда, не светится на фронтенде
- Дать пользователю возможность отозвать/удалить ключ в любой момент

**Результат этапа:** систему можно открыть для других людей — каждый платит за свои вызовы моделей своим ключом, твой бюджет расходуется только на твои собственные задачи.

## Этап 9 — Кэширование запросов для экономии токенов

**Цель:** не платить дважды за то, что уже считалось — похожие/повторяющиеся запросы к LLM обслуживаются из кэша, а не новым вызовом API.

### Где здесь есть что кэшировать

1. **Prompt caching на уровне провайдера** (самый простой и высокий ROI)
   Anthropic API поддерживает кэширование части промпта (system prompt, большой контекст проекта из `PROJECT_CONVENTIONS.md`, проиндексированный код) — при повторных вызовах с одинаковым префиксом промпта платишь за него сильно меньше.
   - [ ] Пометить статичные части промпта (system prompt агента, конвенции проекта, RAG-контекст) как `cache_control: { type: "ephemeral" }` в запросе к Claude API
   - [ ] Это особенно выгодно для Этапа 7 (много задач подряд по одному проекту используют один и тот же большой контекст конвенций)

2. **Кэш на уровне приложения (собственный слой)**
   Для случаев, когда сам запрос (не просто префикс, а целиком) повторяется или очень похож:
   ```sql
   CREATE TABLE llm_cache (
       id UUID PRIMARY KEY,
       prompt_hash VARCHAR,       -- хэш от (system_prompt + user_prompt + model)
       response TEXT,
       tokens_saved INT,
       created_at TIMESTAMP,
       expires_at TIMESTAMP
   );
   ```
   - [ ] Перед вызовом модели — посчитать хэш от промпта, проверить кэш
   - [ ] Если найдено и не устарело (`expires_at`) — вернуть закэшированный ответ, не тратя токены
   - [ ] TTL на кэш (например, 24 часа) — чтобы не отдавать устаревшие ответы для задач, где контекст мог измениться

3. **Semantic caching (продвинутый вариант, опционально)**
   Вместо точного совпадения хэша — искать **похожие** по смыслу запросы через vector similarity (используя уже существующую vector DB из Этапа 7):
   - [ ] Эмбеддинг нового промпта → поиск ближайшего в кэше по косинусному расстоянию
   - [ ] Если схожесть выше порога (например, 95%) — переиспользовать ответ или использовать его как основу вместо генерации с нуля
   - [ ] Это ловит случаи вроде "почти такая же задача, только другое имя переменной"

### Задачи этапа

- [ ] Включить provider-level prompt caching для system prompt'ов агентов и контекста проекта
- [ ] Таблица `llm_cache` + сервис проверки/записи кэша перед каждым вызовом `ModelProvider`
- [ ] Метрика на дашборде: "сэкономлено токенов благодаря кэшу" (отдельный график рядом с расходом бюджета)
- [ ] (Опционально) Semantic caching через vector DB для похожих, но не идентичных запросов

### Зачем это важно для проекта в целом

- Прямая связь с Task Budget (Этап 1) — кэш увеличивает эффективный "запас" бюджета на задачу
- При BYOK (Этап 8) — экономия токенов особенно ценна для пользователя, который платит своими деньгами
- Хороший разговорный пункт на собеседовании: "я не просто трекаю расход токенов, но и активно его оптимизирую через многоуровневое кэширование"

**Результат этапа:** система тратит меньше токенов на повторяющийся/похожий контекст, что напрямую снижает стоимость использования — особенно заметно при повторных задачах над одним и тем же проектом.

## Возможные риски и как их обходить

- **Стоимость экспериментов**: ограничить бюджет на задачу (например, макс. 3 раунда), использовать Claude Haiku для черновых раундов, Sonnet — для финального
- **Безопасность выполнения кода**: агент никогда не должен исполнять код на хосте — только в изолированном контейнере/CI-раннере
- **Сложность MCP-интеграции**: если кастомный MCP-сервер займёт слишком много времени — можно начать с прямых вызовов GitHub REST API и добавить MCP-обвязку позже, когда ядро уже работает
